How should we scaffold this?

Three approaches — pick the one that fits how you want to build

A

Minimal Foundation

Next.js app + NextAuth + Neon schema. Auth and database only. Get something working fast, build the rest on top.

quote-tracker/
├── app/
│   ├── (auth)/login/
│   ├── dashboard/
│   └── api/auth/
├── lib/
│   ├── db.ts
│   └── auth.ts
└── drizzle/ (schema + migrations)

Pros

  • Fastest to get running
  • Low complexity upfront

Cons

  • Cloudflare Worker not included
  • No thought given to future modules yet
B

Full Week 1 Scaffold

Everything in A, plus Cloudflare Worker skeleton and ServiceM8 email template. Complete Week 1 code in one session.

quote-tracker/
├── app/
│   ├── (auth)/login/
│   ├── dashboard/
│   └── api/auth/
├── lib/
│   ├── db.ts
│   └── auth.ts
├── workers/ (Cloudflare Worker)
│   └── tracker/index.ts
├── drizzle/ (schema + migrations)
└── templates/ (ServiceM8 email)

Pros

  • Week 2 can start immediately
  • All the code pieces in one go

Cons

  • More to set up in one session
  • Still no structure for future modules
C

Platform-Aware Scaffold ★ Recommended

Everything in B, but structured from day one as the rgtools.co.nz platform — not just a quote tracker. No restructuring needed when Calculator, PS1/PS3, and other modules arrive.

quote-tracker/
├── app/
│   ├── (auth)/login/
│   ├── (dashboard)/
│   │   └── quote-tracker/ ← module route
│   └── api/
├── modules/
│   ├── quote-tracker/ ← module logic
│   └── _shared/ ← future modules share this
├── lib/
│   ├── db.ts ← shared DB client
│   └── auth.ts ← shared auth
├── workers/
│   └── tracker/index.ts
├── drizzle/
└── templates/

Pros

  • README says this is a platform — build it that way
  • Auth, DB, types shared across modules
  • No painful restructuring later

Cons

  • Slightly more upfront structure to set up
